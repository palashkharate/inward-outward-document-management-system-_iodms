## Abstract

This project presents the design and implementation of an Artificial Intelligence-based Predictive Maintenance System for estimating the Remaining Useful Life (RUL) of aircraft turbofan engines using the NASA Commercial Modular Aero-Propulsion System Simulation (C-MAPSS) benchmark dataset. The objective of the project is to develop a data-driven maintenance framework capable of predicting engine degradation before failure, thereby supporting timely maintenance decisions and improving aircraft reliability.

The proposed methodology includes data preprocessing, exploratory data analysis, feature engineering, and machine learning-based regression modelling. Several regression algorithms, namely Linear Regression, Decision Tree Regression, Random Forest Regression, Support Vector Regression (SVR), and Extreme Gradient Boosting (XGBoost), are studied and comparatively evaluated using Root Mean Squared Error (RMSE), Mean Absolute Error (MAE), Coefficient of Determination (R² Score), and NASA Scoring Function. The comparative analysis enables the identification of the most suitable regression model for Remaining Useful Life prediction.

To improve the practical interpretation of the NASA C-MAPSS dataset, representative engineering proxies are discussed for selected anonymous sensor channels, illustrating how similar measurements could be utilized within real aircraft engine health monitoring systems. These mappings are conceptual and intended solely for demonstrating potential deployment scenarios.

An interactive Streamlit dashboard was developed to visualize engine degradation trends, compare model performance, estimate Remaining Useful Life, and generate maintenance recommendations. The developed system demonstrates the potential of Artificial Intelligence in predictive maintenance by enabling condition-based maintenance strategies that can reduce operational costs, minimize unexpected failures, and improve aircraft availability and safety.

Overall, the project demonstrates the practical application of machine learning techniques for aerospace predictive maintenance while providing a scalable framework that can be extended to real-world aircraft health monitoring systems.

## 1.1 Introduction

Aircraft engines operate under extreme mechanical and thermal conditions, making regular maintenance essential for ensuring safety and reliability. Traditional maintenance approaches, such as reactive and preventive maintenance, often lead to either unexpected failures or unnecessary servicing. With the advancement of Artificial Intelligence (AI) and Machine Learning (ML), predictive maintenance has emerged as an efficient alternative by estimating the Remaining Useful Life (RUL) of engine components before failure occurs.

Predictive maintenance analyses historical operational data to identify degradation patterns and predict future equipment health. This enables maintenance to be scheduled only when required, reducing downtime, maintenance costs, and operational risks.

This project develops an AI-based predictive maintenance system using the NASA C-MAPSS aircraft engine dataset. Multiple machine learning regression algorithms are implemented and compared to predict the Remaining Useful Life of aircraft engines. An interactive Streamlit dashboard is also developed to visualize predictions and assist in maintenance decision-making. The proposed system demonstrates how machine learning can support intelligent aircraft maintenance and improve operational efficiency.

## 1.2 Aircraft Engine Health Monitoring

Aircraft Engine Health Monitoring (EHM) is the continuous process of monitoring an aircraft engine's condition using operational data to ensure safe and reliable performance. Modern aircraft engines operate under extreme temperature, pressure, and mechanical stress, causing gradual degradation of engine components over time. Continuous monitoring helps detect early signs of wear and prevents unexpected failures.

Aircraft engines are equipped with numerous sensors that collect operational information during every flight cycle. These measurements provide valuable insights into engine performance and degradation. By analysing this data, maintenance engineers can identify abnormal operating conditions and make informed maintenance decisions before critical failures occur.

Traditional maintenance methods rely on scheduled inspections or repairs after failure, which may lead to unnecessary maintenance or unexpected breakdowns. Engine Health Monitoring overcomes these limitations by continuously evaluating engine condition and supporting data-driven maintenance strategies. When integrated with Artificial Intelligence and Machine Learning, EHM enables prediction of the Remaining Useful Life (RUL) of engine components, allowing maintenance to be performed at the optimal time.

In this project, the NASA C-MAPSS dataset is used to simulate the aircraft engine health monitoring process. Machine learning algorithms analyse historical engine operational data to estimate Remaining Useful Life, demonstrating how intelligent monitoring systems can improve maintenance efficiency, reduce downtime, and enhance aircraft safety.

## 1.4 Maintenance Strategies

Aircraft maintenance strategies have evolved from corrective approaches to intelligent predictive systems. The four major maintenance strategies are Reactive Maintenance, Preventive Maintenance, Condition-Based Maintenance (CBM), and Predictive Maintenance (PdM).

Reactive Maintenance is performed only after equipment failure occurs, resulting in high repair costs and unexpected downtime.

Preventive Maintenance follows predefined maintenance schedules regardless of the equipment's actual condition. Although it reduces sudden failures, it may lead to unnecessary replacement of healthy components.

Condition-Based Maintenance (CBM) continuously monitors equipment health using sensor data. Maintenance is initiated only when abnormal conditions are detected, reducing unnecessary servicing.

Predictive Maintenance uses Artificial Intelligence and Machine Learning to analyse historical operational data and estimate the Remaining Useful Life of components. It enables maintenance before failure occurs, improving reliability while minimizing maintenance cost and downtime.

## 2.1 Introduction

The aerospace industry has increasingly adopted advanced technologies to enhance safety, efficiency, and operational reliability. Predictive maintenance, powered by Artificial Intelligence (AI) and Machine Learning (ML), has emerged as a transformative approach to manage the health of complex systems like aircraft turbofan engines. ## 2.2 NASA C-MAPSS Dataset Overview

The NASA Commercial Modular Aero-Propulsion System Simulation (C-MAPSS) dataset is a widely recognized benchmark for prognostics and health management (PHM) research, particularly for aircraft turbofan engines. It provides simulated multivariate time-series sensor data that captures engine degradation under various operating conditions and fault modes. The dataset is crucial for developing and evaluating RUL prediction algorithms due to its realistic representation of engine wear and tear.

Each dataset (FD001, FD002, FD003, FD004) within C-MAPSS represents different operating conditions and fault modes, offering a diverse challenge for RUL prediction models. The data includes operational settings, sensor measurements, and the true RUL for each engine, allowing for robust model training and evaluation.

## 2.3 Literature Review

The prediction of Remaining Useful Life (RUL) for aircraft turbofan engines has been a significant area of research, with numerous studies exploring various machine learning and deep learning approaches. Early research often focused on statistical methods and physics-based models, but the advent of large datasets and computational power has shifted the focus towards data-driven techniques.

Studies have shown the effectiveness of traditional machine learning algorithms such as Support Vector Machines (SVM), Random Forests, and Gradient Boosting Machines (GBM) in capturing complex degradation patterns. For instance, [Reference 1] utilized SVMs for RUL prediction, demonstrating their capability to handle non-linear relationships in sensor data. Ensemble methods like Random Forests [Reference 2] and XGBoost [Reference 3] have also proven highly effective due to their ability to combine multiple weak learners into a strong predictive model, often outperforming single models.

Deep learning architectures, particularly Recurrent Neural Networks (RNNs) and Long Short-Term Memory (LSTM) networks, have gained prominence for their ability to process sequential data inherent in time-series sensor readings. [Reference 4] demonstrated that LSTMs can effectively learn long-term dependencies in engine degradation data, leading to improved RUL predictions. Convolutional Neural Networks (CNNs) have also been adapted for RUL prediction by treating time-series data as images or using 1D convolutions to extract features [Reference 5].

Hybrid approaches, combining the strengths of different models or integrating feature engineering with deep learning, are also a growing trend. For example, [Reference 6] proposed a hybrid model that uses a CNN for feature extraction followed by an LSTM for sequence learning, achieving state-of-the-art performance on the C-MAPSS dataset.

## 2.4 Comparison of Machine Learning Algorithms

The choice of machine learning algorithm for RUL prediction is critical and depends on various factors, including data characteristics, computational resources, and desired interpretability. Traditional machine learning models like Random Forest, XGBoost, and SVR offer a balance of performance and interpretability. Random Forest, an ensemble of decision trees, is robust to overfitting and can handle high-dimensional data. XGBoost, a gradient boosting framework, is known for its speed and accuracy, often winning machine learning competitions. Support Vector Regression (SVR) is effective in high-dimensional spaces and can model non-linear relationships using kernel tricks.

Deep learning models, while often achieving higher accuracy, typically require larger datasets and more computational power. Their black-box nature can also be a disadvantage in safety-critical applications where model interpretability is crucial. However, advancements in explainable AI (XAI) are addressing this limitation.

## 2.5 Research Gap

Despite significant advancements in RUL prediction, several research gaps remain. One key area is the development of robust models that can generalize across diverse operating conditions and fault modes without extensive retraining. Many existing models perform well on specific subsets of the C-MAPSS dataset but struggle with unseen variations. Furthermore, the integration of multi-source data (e.g., maintenance logs, flight data, environmental factors) beyond sensor readings remains an area for further exploration to enhance prediction accuracy and provide a more holistic view of engine health.

Another gap lies in the real-time deployment and operationalization of these models within existing avionics systems, especially considering the stringent certification requirements and computational constraints of embedded systems. The development of lightweight, yet accurate, models suitable for edge deployment is crucial. Finally, enhancing the interpretability and explainability of complex machine learning models for RUL prediction is vital for gaining trust from maintenance engineers and facilitating informed decision-making in safety-critical aerospace applications.

# 3. CHAPTER 3 – DATASET ANALYSIS AND METHODOLOGY

## 3.1 Introduction

This chapter details the methodology employed for the AI-based predictive maintenance system, encompassing the dataset analysis, preprocessing steps, feature engineering techniques, and the proposed machine learning approach for Remaining Useful Life (RUL) prediction. The objective is to provide a clear and reproducible framework for developing robust RUL prediction models.

## 3.2 NASA C-MAPSS Dataset

The NASA Commercial Modular Aero-Propulsion System Simulation (C-MAPSS) dataset is a collection of simulated aircraft turbofan engine degradation data. It comprises four distinct subsets (FD001, FD002, FD003, FD004), each representing different operating conditions and fault modes. Each dataset includes:

*   **Unit Number:** Unique identifier for each engine.
*   **Cycle:** Time in cycles (or operational units) for each engine.
*   **Operational Settings:** Three operational settings that vary during flight.
*   **Sensor Measurements:** 21 sensor measurements that capture engine performance and degradation.
*   **Remaining Useful Life (RUL):** The true RUL for each engine, indicating the number of cycles remaining before failure.

The dataset is characterized by run-to-failure trajectories, where engines start with varying degrees of initial wear and degrade until failure. This makes it ideal for training and evaluating RUL prediction models.

## 3.3 Data Preprocessing

Effective data preprocessing is crucial for enhancing the performance of machine learning models. The following steps were applied to the C-MAPSS dataset:

1.  **RUL Capping:** The true RUL values were capped at a maximum threshold (e.g., 125 cycles). This is based on the assumption that maintenance decisions are typically made within a certain window before failure, and predicting RUL far in advance might not be as critical or accurate.
2.  **Feature Normalization:** Sensor measurements and operational settings were normalized using Min-Max scaling. This ensures that all features contribute equally to the model training process and prevents features with larger values from dominating the learning algorithm.
3.  **Feature Selection:** Initial analysis and domain knowledge were used to select the most relevant sensor measurements and operational settings, discarding those with little variance or direct correlation to degradation.

## 3.4 Feature Engineering

Feature engineering is the process of creating new features from existing raw data to improve the predictive power of machine learning models. For the C-MAPSS dataset, temporal features were engineered to capture the degradation trends:

1.  **Rolling Statistics:** Moving averages, standard deviations, minimums, and maximums of sensor measurements were calculated over a defined window size (e.g., 10-20 cycles). These features provide insights into the recent history and variability of engine performance.
2.  **Polynomial Features:** Polynomial transformations of certain sensor readings were explored to capture non-linear relationships with RUL.
3.  **Interaction Features:** Combinations of existing features were created to capture synergistic effects that might influence engine degradation.

## 3.5 Proposed Methodology

The proposed methodology for RUL prediction involves a supervised learning approach. After preprocessing and feature engineering, the dataset is split into training and testing sets. Machine learning regression models are then trained on the engineered features to predict the RUL. The overall workflow is as follows:

1.  **Data Acquisition:** Load the NASA C-MAPSS training and test datasets.
2.  **Preprocessing:** Apply RUL capping, normalization, and feature selection.
3.  **Feature Engineering:** Generate rolling statistics and other temporal features.
4.  **Model Training:** Train selected machine learning models (Random Forest, XGBoost, SVR) on the preprocessed and engineered training data.
5.  **Model Evaluation:** Evaluate model performance on the test set using various metrics.
6.  **Deployment:** Deploy the best-performing model via an interactive Streamlit dashboard.

## 3.6 Evaluation Metrics

The performance of the RUL prediction models was evaluated using the following metrics:

1.  **Root Mean Squared Error (RMSE):** Measures the average magnitude of the errors. Lower RMSE indicates better accuracy.
2.  **Mean Absolute Error (MAE):** Measures the average absolute difference between predicted and actual values. Less sensitive to outliers than RMSE.
3.  **R-squared (R²):** Represents the proportion of the variance in the dependent variable that is predictable from the independent variables. Higher R² indicates a better fit.
4.  **NASA PHM08 Scoring Function:** A domain-specific metric that asymmetrically penalizes late predictions (overestimation of RUL) ten times more heavily than early predictions (underestimation of RUL). A lower NASA score is desirable, reflecting safer predictions.

## 5.2 Performance Comparison of Machine Learning Models

The performance of the trained machine learning models (Random Forest, XGBoost, and Support Vector Regression) for Remaining Useful Life (RUL) prediction was rigorously evaluated using a combination of standard regression metrics and the domain-specific NASA PHM08 Scoring Function. The evaluation was primarily conducted on the FD001 subset of the NASA C-MAPSS dataset, which represents a single operating condition and a single fault mode, providing a clear baseline for comparison.

### 5.2.1 Quantitative Evaluation

The following table summarizes the key performance metrics for each model on the FD001 subset:

| Model           | RMSE   | MAE    | R²      | NASA Score |
| :-------------- | :----- | :----- | :------ | :--------- |

From the results, it is evident that the **Support Vector Regression (SVR)** model achieved the best performance across all evaluated metrics for the FD001 subset. The R² score of 0.9129 suggests that SVR explains over 91% of the variance in the RUL, demonstrating a strong fit to the degradation patterns.

### 5.2.2 Analysis of NASA PHM08 Scoring Function

The NASA PHM08 Scoring Function is particularly insightful for aerospace applications due to its asymmetric penalty for prediction errors. A lower NASA Score indicates better performance. The SVR model achieved the lowest NASA Score of 8,462, reinforcing its superior performance. This metric is crucial because it heavily penalizes late predictions (overestimating RUL) ten times more than early predictions (underestimating RUL). In a safety-critical domain like aviation, overestimating the remaining life of an engine can lead to catastrophic failures, making the ability to avoid such errors highly valued. The SVR model's lower score suggests it is more effective at minimizing these high-penalty late predictions.

### 5.2.3 Comparative Discussion

*   **Random Forest:** Served as a robust baseline, demonstrating good predictive capabilities with an R² of over 0.90. Its ensemble nature provides stability and interpretability.
*   **XGBoost:** Showed an improvement over Random Forest, which is expected given its advanced gradient boosting technique that sequentially corrects errors. Its performance was close to SVR but slightly higher in RMSE and NASA Score.
*   **SVR:** Outperformed both tree-based models. This can be attributed to its ability to effectively model complex, non-linear relationships in the high-dimensional feature space created by the feature engineering process, particularly with the RBF kernel. The careful normalization of features also played a significant role in SVR's superior performance.

These results highlight the effectiveness of the chosen preprocessing and feature engineering steps in preparing the C-MAPPS data for machine learning models. The SVR model, in particular, demonstrated a strong capability to predict RUL accurately and safely, making it a promising candidate for real-world predictive maintenance systems.

## 6.2 Challenges Faced

Both the Predictive Maintenance project and the IODMS implementation presented distinct challenges, requiring innovative solutions and a robust problem-solving approach. Addressing these challenges was crucial for the successful completion of the internship objectives.

### 6.2.1 Challenges in Predictive Maintenance Project

*   **Data Quality and Preprocessing:** The raw NASA C-MAPSS dataset, while comprehensive, required extensive preprocessing. This included handling missing values, noise reduction, and feature scaling. The primary challenge was to derive meaningful features from the time-series sensor data that accurately captured engine degradation patterns. **Solution:** Implementation of a robust preprocessing pipeline involving moving average filters, normalization techniques, and the creation of new features such as rolling statistics (mean, standard deviation, min, max) over defined window sizes. The RUL capping strategy was also critical to focus the model on the most relevant degradation phase.
*   **Model Selection and Optimization:** Choosing the optimal machine learning model from a range of algorithms (Random Forest, XGBoost, SVR) and fine-tuning their hyperparameters was challenging. The goal was to achieve high accuracy while maintaining interpretability and computational efficiency. **Solution:** A systematic approach involving cross-validation and grid search (or similar hyperparameter optimization techniques) was employed. The NASA PHM08 scoring function was used as a primary evaluation metric to guide model selection, prioritizing models that minimized the severe penalty for late RUL predictions.
*   **Generalization Gap:** Ensuring that the models generalized well from the training data to unseen test data, especially across different operating conditions and fault modes within the C-MAPSS dataset, was a significant concern. **Solution:** Careful splitting of the dataset to ensure representative training and testing sets, along with rigorous evaluation on held-out test data, helped assess and improve generalization capabilities. Techniques like ensemble methods (Random Forest, XGBoost) inherently offer better generalization due to their averaging of multiple decision trees.

### 6.2.2 Challenges in IODMS Implementation

*   **Air-Gapped Environment Deployment:** Deploying a full-stack application in an air-gapped LAN without internet access was the most significant challenge. This meant no direct access to package managers, Docker registries, or online documentation. **Solution:** A comprehensive offline deployment strategy was developed. This involved pre-downloading all necessary software packages, Docker images, and dependencies on an internet-connected machine, securely transferring them via physical media, and then performing offline installations and Docker image loading on the target server. This ensured all components were available within the isolated network.
*   **Legacy System Migration:** The IODMS was a redevelopment of an inaccessible Microsoft Access-based system, with only exported XLSX files of five tables available for one-time migration. Additionally, over 5000 scanned documents needed to be migrated into a new folder structure. **Solution:** A data migration plan was devised to parse the XLSX data and import it into the new PostgreSQL database. For the scanned documents, a script was planned to organize them into the new `IODMS/Inward/{Year}/{Folder ID}/` structure, ensuring proper naming conventions and metadata association.
*   **Security and Compliance:** Adhering to HAL's stringent security requirements, including the Official Secrets Act, 1923, necessitated robust security features and careful implementation. **Solution:** Implementation of Role-Based Access Control (RBAC), secure session management using `sessionStorage`, and the development of a highly restricted Auditor View with features like disabled text selection and watermarking. All development adhered to secure coding practices to minimize vulnerabilities.

## Appendix E: Docker Deployment Workflow

This appendix details the Docker-based deployment workflow for the Inward Outward Document Management System (IODMS), with a particular focus on the strategies employed for successful deployment within an air-gapped local area network (LAN) environment. The air-gapped nature of the deployment necessitated a meticulous, offline-first approach to ensure all dependencies and container images were available without internet access.

### E.1 Offline Docker Image Management

In an air-gapped environment, direct access to public Docker registries (like Docker Hub) is impossible. Therefore, a multi-step process was adopted for managing Docker images:

1.  **Image Pull on Internet-Connected Machine:** All necessary Docker images for the IODMS application (e.g., PostgreSQL, FastAPI backend, React frontend) were identified and pulled from public registries on a secure, internet-connected machine.
2.  **Image Saving:** The pulled Docker images were then saved as `.tar` archives using the `docker save` command. This bundles the image layers and metadata into a single file.
    ```bash
    docker save -o postgres_image.tar postgres:13
    docker save -o backend_image.tar iodms_backend:latest
    docker save -o frontend_image.tar iodms_frontend:latest
    ```
3.  **Secure Transfer:** The `.tar` image files were securely transferred to the air-gapped server using physical media (e.g., encrypted USB drives).
4.  **Image Loading on Air-Gapped Server:** On the air-gapped server, the transferred `.tar` files were loaded into the local Docker daemon using the `docker load` command.
    ```bash
    docker load -i postgres_image.tar
    docker load -i backend_image.tar
    docker load -i frontend_image.tar
    ```
This process ensured that all required container images were available locally on the air-gapped server, ready for deployment.

### E.2 Docker Compose for Service Orchestration

Docker Compose was utilized to define and run the multi-container IODMS application. A `docker-compose.yml` file orchestrated the services, networks, and volumes, ensuring that the frontend, backend, and database containers were correctly configured and interconnected.

**[Placeholder for a simplified `docker-compose.yml` snippet]**

*Figure E.1: Simplified Docker Compose Configuration for IODMS*

Key aspects of the Docker Compose configuration included:

*   **Service Definitions:** Each component (frontend, backend, database) was defined as a separate service.
*   **Network Configuration:** Custom bridge networks were used to facilitate secure communication between services.
*   **Volume Mounting:** Persistent volumes were configured for the PostgreSQL database to ensure data persistence across container restarts.
*   **Environment Variables:** Sensitive configurations and database credentials were managed through environment variables, often loaded from a `.env` file.

### E.3 Deployment Steps on Air-Gapped Server

The overall deployment process on the air-gapped server involved the following steps:

1.  **Prerequisites Installation:** Ensure Docker and Docker Compose are installed. For an air-gapped environment, these would also need to be installed offline from pre-downloaded packages.
2.  **Transfer Project Files:** Copy the entire IODMS project directory (including the `docker-compose.yml` and any necessary build contexts) to the air-gapped server.
3.  **Load Docker Images:** Execute the `docker load` commands as described in Section E.1 to make all required images available.
4.  **Database Seeding:** Run the `seed.py` script (or a similar database initialization script) to populate the PostgreSQL database with initial master data and an admin user. This is crucial for the application's first run.
    ```bash
    python3 seed.py
    ```
5.  **Start Services:** Navigate to the project root directory and start the application using Docker Compose.
    ```bash
    docker compose up -d
    ```
    The `-d` flag runs the containers in detached mode, allowing them to run in the background.
6.  **Verification:** Verify that all services are running correctly and accessible within the LAN.

This robust Docker-based deployment strategy, adapted for air-gapped environments, ensures the IODMS can be reliably deployed and operated in highly secure and isolated network infrastructures.
